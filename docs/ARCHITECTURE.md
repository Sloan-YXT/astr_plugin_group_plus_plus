## 一句话概括

> 本插件**完全接管了 AI 的上下文构建**，它不依赖 AstrBot 平台默认的历史消息和提示词系统，而是自己从头构建发送给 AI 的所有内容（历史消息、系统提示词、当前消息），并在最后一刻替换掉平台准备好的上下文。

---

## 为什么需要自建上下文？

AstrBot 平台默认的上下文构建是通用的——它把平台收到的历史消息、系统提示词、用户消息直接传给 AI。但对于一个需要"读空气"的群聊插件来说，这远远不够：

| 需求 | 平台默认行为 | 本插件的做法 |
|------|-------------|-------------|
| 消息格式 | 原样传递 | 添加时间戳、发送者名称和ID |
| 历史管理 | 平台统一管理 | 插件自己存储，可独立清除、截断 |
| Bot 自己的回复 | 混在历史中 | 标记为 `【禁止重复-你的历史回复】` |
| 未回复的消息 | 不区分 | 标记为 `【📦近期未回复】`，提供上下文 |
| 当前消息 | 与历史混在一起 | 用分隔线明确区分"历史"和"当前" |
| 系统指令 | 平台注入 | 插件注入自己的读空气/回复指令 |
| 窗口缓冲消息 | 不支持 | 等待窗口收集的后续消息单独标注 |

## 核心架构：标记 → 传递 → 拦截 → 替换

整个机制可以用四个字概括：**"偷天换日"**。

│                                                                  │

│  ┌──────────────────────────────────────┐                       │
│  │ ReplyHandler.generate_reply()        │                       │
│  │                                      │                       │
│  │  构建 full_prompt = 完整格式化历史   │                       │
│  │  + 系统指令 + 当前消息 + 记忆       │                       │
│  │                                      │                       │
│  │  但是！不能直接发给AI               │                       │
│  │  因为还要经过平台的 Hook 链         │                       │
│  └──────────────┬───────────────────────┘                       │
│                 ↓                                                │
│  ② 把完整上下文"藏"在标记中                                    │
│  ┌──────────────────────────────────────┐                       │
│  │ event.set_extra() 设置标记：         │                       │
│  │                                      │                       │
│  │  🏷️ PLUGIN_REQUEST_MARKER = True    │ ← 标记"这是我的请求" │
│  │  🏷️ PLUGIN_CUSTOM_PROMPT = 完整上下文│ ← 藏好完整上下文     │
│  │  🏷️ PLUGIN_CUSTOM_SYSTEM_PROMPT     │ ← 人格提示词         │
│  │  🏷️ PLUGIN_IMAGE_URLS              │ ← 图片               │
│  │  🏷️ PLUGIN_FUNC_TOOL               │ ← 工具集             │
│  └──────────────┬───────────────────────┘                       │
│                 ↓                                                │
│  ③ 用"短消息"调用平台 API                                      │
│  ┌──────────────────────────────────────┐                       │
│  │ event.request_llm(                   │                       │
│  │   prompt = "短消息"  ← 只传当前消息 │                       │

│  │ )                                    │                       │
│  └──────────────┬───────────────────────┘                       │
│                 ↓                                                │
│  ④ 平台 Hook 链依次执行                                        │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │                                                          │   │
│  │  优先级 100000  emotionai 插件                          │   │
│  │  ├─ 在 system_prompt 中追加情绪规则                     │   │
│  │  ↓                                                      │   │
│  │  优先级 0       AstrBot 平台 LTM                        │   │
│  │  ├─ 在 system_prompt 中追加群聊历史                     │   │
│  │  │   "You are now in a chatroom..."                     │   │
│  │  ├─ 填充 req.contexts = [平台对话上下文]               │   │
│  │  ↓                                                      │   │
│  │  优先级 -1      ★ 本插件的 on_llm_request ★           │   │
│  │  ├─ 检查标记：是我的请求吗？→ 是！                     │   │
│  │  ├─ 提取其他插件追加的内容（保留 emotionai 等）        │   │
│  │  ├─ 删除平台 LTM 注入的群聊历史                        │   │
│  │  ├─ ★ 用藏好的完整上下文替换 req.prompt ★             │   │
│  │  ├─ ★ 清空 req.contexts ★                             │   │
│  │  ├─ 重组 system_prompt = 人格 + 其他插件内容           │   │
│  │  └─ 清除所有标记                                       │   │
│  │                                                          │   │
│  └──────────────────────────────────────────────────────────┘   │
│                 ↓                                                │
│  ⑤ 最终发送给 AI 的内容                                        │
│  ┌──────────────────────────────────────┐                       │
│  │ system_prompt:                       │                       │
│  │   人格设定                           │                       │
│  │   + emotionai 情绪规则（保留）       │                       │
│  │   + 其他兼容插件的内容（保留）       │                       │
│  │                                      │                       │
│  │ prompt:                              │                       │
│  │   ★ 插件自己格式化的完整上下文 ★   │                       │
│  │   （历史消息 + 系统指令 + 当前消息） │                       │
│  │                                      │                       │

│  └──────────────────────────────────────┘                       │
└─────────────────────────────────────────────────────────────────┘

## 什么被替换了？什么被保留了？

### 被完全替换（平台原始内容被丢弃）

| 内容 | 平台原始值 | 插件替换为 |
|------|-----------|-----------|
| `req.prompt` | 用户发送的短消息 | 插件格式化的**完整上下文**（历史+指令+当前消息） |
| `req.contexts` | 平台对话历史数组 | `[]`（空数组，不使用） |
| 平台 LTM 群聊历史 | "You are now in a chatroom..." | **删除**（被插件自己的历史替代） |
| `req.func_tool` | 平台默认工具集 | 插件自定义工具集 |

### 被保留（其他插件的贡献不受影响）

| 内容 | 来源 | 保留方式 |
|------|------|----------|
| emotionai 情绪规则 | emotionai 插件 | 从 system_prompt 中提取并保留 |
| 好感度规则 | FavourPro 等插件 | 从 system_prompt 中提取并保留 |
| 学习规则 | SelfLearning 等插件 | 从 system_prompt 中提取并保留 |
| 人格设定 | AstrBot 平台 | 作为 system_prompt 基础 |

**保留机制的原理**：插件知道自己传入的 `system_prompt` 是什么（人格设定），所以在 `on_llm_request` 中它把当前 `req.system_prompt` 减去自己传入的部分，剩下的就是其他插件追加的内容，然后再用正则删除平台 LTM 注入的群聊历史部分，最后把"自己的人格 + 其他插件的内容"重新组装。

## 消息存储：双轨制

本插件维护了一套**独立于平台的消息存储系统**：

```
┌─────────────────────────────────────────────────────┐
│              消息存储双轨制                           │
├─────────────────────────────────────────────────────┤
│                                                      │
│  轨道 1：插件自定义存储（主存储）                    │
│  ┌──────────────────────────────────────────┐       │
│  │ custom_storage（JSON 文件）              │       │
│  │                                          │       │
│  │ • 插件自己管理的消息历史                 │       │
│  │ • 格式化后的消息（含时间戳、发送者）     │       │
│  │ • 经过 MessageCleaner 清洗的内容        │       │
│  │ • 受 custom_storage_max_messages 限制   │       │
│  │ • 受 gcp_reset_here 历史截止时间戳控制  │       │
│  │ • 用于：构建发送给 AI 的历史上下文      │       │
│  └──────────────────────────────────────────┘       │
│                                                      │
│  轨道 2：平台官方存储（同步写入）                    │
│  ┌──────────────────────────────────────────┐       │
│  │ platform_message_history（SQLite 数据库）│       │
│  │                                          │       │
│  │ • AstrBot 平台的官方历史表               │       │
│  │ • 插件也会写入（保持平台数据一致）      │       │
│  │ • 但读取时使用自定义存储而非此表        │       │
│  │ • /reset 只清此表但不影响插件存储       │       │
│  │ • gcp_reset 会同时处理两个轨道          │       │
│  └──────────────────────────────────────────┘       │
│                                                      │
│  待处理池（临时缓存）                                │
│  ┌──────────────────────────────────────────┐       │
│  │ pending_messages_cache                   │       │
│  │                                          │       │
│  │ • 未通过概率筛选的消息暂存于此          │       │
│  │ • 下次回复时作为"近期未回复"上下文合并  │       │
│  │ • 受 pending_cache_max_count 限制       │       │
│  │ • 受 pending_cache_ttl_seconds 过期控制 │       │
│  └──────────────────────────────────────────┘       │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### 消息保存时的清洗

消息在保存到自定义存储前，会经过 `MessageCleaner` 清洗，**移除所有系统提示词**：

```
保存前（AI 看到的）：
  "[系统提示]注意，现在有人在直接@你说话
   [2026-03-13 周四 14:30:00] 张三(ID:111): 你好啊
   === 背景信息 ===
   💭 相关记忆：xxx"

保存后（存入历史的）：
  "张三(ID:111): 你好啊"
```

这确保了下次读取历史时不会出现嵌套的系统提示词。

## 决策 AI vs 回复 AI：两条独立的 AI 调用路径

本插件有**两次独立的 AI 调用**，它们走完全不同的路径：

```
┌──────────────────────────────────┐   ┌──────────────────────────────────┐
│     决策 AI（"读空气"）          │   │     回复 AI（"生成回复"）        │
├──────────────────────────────────┤   ├──────────────────────────────────┤
│                                  │   │                                  │
│ 调用方式：                       │   │ 调用方式：                       │
│   provider.text_chat()           │   │   event.request_llm()            │
│   ↑ 直接调用 AI 提供商           │   │   ↑ 走平台标准流程               │
│   ↑ 不经过任何 Hook              │   │   ↑ 经过 Hook 链                 │
│   ↑ 不受其他插件影响             │   │   ↑ 受本插件 on_llm_request 拦截 │
│                                  │   │                                  │
│ 系统提示词：                     │   │ 系统提示词：                     │
│   内置的决策判断指令             │   │   人格设定 + 其他插件内容        │
│   + decision_ai_extra_prompt     │   │   + 回复指令                     │
│   + 注意力/情绪/疲劳状态        │   │   + reply_ai_extra_prompt        │
│                                  │   │                                  │
│ 上下文：                         │   │ 上下文：                         │
│   近期消息摘要（较短）           │   │   完整格式化历史（完整）         │
│                                  │   │   + 记忆 + 工具 + 窗口消息      │
│                                  │   │                                  │
│ 输出：                           │   │ 输出：                           │
│   YES / NO                       │   │   AI 的回复文本                  │
│                                  │   │                                  │
│ AI 提供商：                      │   │ AI 提供商：                      │
│   decision_ai_provider_id        │   │   AstrBot 默认提供商             │
│   （可单独配置轻量模型）         │   │                                  │
│                                  │   │                                  │
└──────────────────────────────────┘   └──────────────────────────────────┘
```

## 与其他插件的兼容性

### 兼容的插件类型

以下类型的插件**可以与本插件共存**：

| 插件类型 | 原理 | 示例 |
|----------|------|------|
| 只修改 system_prompt 的插件 | 追加的内容会被保留 | emotionai、FavourPro、SelfLearning |
| 使用独立 AI 调用的插件 | 互不干扰 | 记忆插件（通过 provider.text_chat 调用） |
| 不修改上下文的插件 | 没有冲突 | 重启插件、工具插件 |

### 可能冲突的插件类型

以下类型的插件**可能与本插件产生冲突**：

| 插件类型 | 冲突原因 | 后果 |
|----------|----------|------|
| 修改 `req.contexts` 的插件 | 本插件会清空 contexts | 该插件注入的上下文会丢失 |
| 修改 `req.prompt` 的插件 | 本插件会替换 prompt | 该插件对 prompt 的修改会丢失 |
| 有自己的 LTM/历史管理的插件 | 两套历史系统冲突 | AI 可能看到重复或矛盾的历史 |
| 依赖平台默认上下文格式的插件 | 格式被完全替换 | 该插件无法正常工作 |

### 冲突判断流程图

```
你想使用的插件修改了什么？
    │
    ├─ 只修改 system_prompt？
    │   └─ ✅ 兼容，内容会被保留
    │
    ├─ 修改 req.contexts？
    │   └─ ❌ 冲突，内容会被清空
    │
    ├─ 修改 req.prompt？
    │   └─ ❌ 冲突，内容会被替换
    │
    ├─ 有自己的历史/上下文管理？
    │   └─ ⚠️ 可能冲突，需要测试
    │
    └─ 都不修改？
        └─ ✅ 兼容，无影响
```

### 如何判断你的其他插件是否兼容？

1. **查看该插件是否注册了 `on_llm_request` Hook**
   - 如果没有 → 兼容
   - 如果有，查看它修改了 `req` 的哪些字段

2. **查看 Hook 优先级**
   - 优先级 > -1（在本插件之前执行）：
     - 修改 `system_prompt` → 兼容（会被保留）
     - 修改 `contexts` 或 `prompt` → 冲突（会被覆盖）
   - 优先级 < -1（在本插件之后执行）：
     - 修改任何内容 → 可能有意外效果

3. **最简单的测试方法**
   - 开启 `enable_debug_log`，查看日志中 AI 收到的实际上下文内容

## Hook 执行顺序详解

AstrBot 的 Hook 系统按**优先级从高到低**执行。本插件使用 `priority=-1`，确保在**最后执行**，这样它可以看到所有其他插件追加的内容，并选择性保留：

```
on_llm_request Hook 执行链
═════════════════════════════════════════════════

优先级 100000   emotionai
                ├─ 在 system_prompt 末尾追加情绪规则
                └─ 不修改 contexts 和 prompt
                    ↓
优先级 0        AstrBot 平台 LTM（长期记忆）
                ├─ 在 system_prompt 末尾追加：
                │   "You are now in a chatroom.
                │    The chat history is as follows:
                │    [平台存储的群聊历史]"
                ├─ 设置 req.contexts = [平台对话上下文]
                └─ 不修改 prompt
                    ↓
优先级 -1       ★ 本插件 on_llm_request ★
                │
                │  此时 req.system_prompt 的内容是：
                │  "[人格] + [emotionai 规则] + [LTM 群聊历史]"
                │
                ├─ 1. 从 system_prompt 中提取出"人格"部分
                │     （因为插件知道自己传入的人格内容）
                │
                ├─ 2. 计算差异 = 当前 system_prompt - 人格
                │     得到 "[emotionai 规则] + [LTM 群聊历史]"
                │
                ├─ 3. 用正则删除 LTM 的群聊历史部分
                │     剩下 "[emotionai 规则]"
                │
                ├─ 4. 重组 system_prompt：
                │     "[人格] + [emotionai 规则]"
                │
                ├─ 5. 替换 req.prompt = 插件的完整上下文
                │
                ├─ 6. 清空 req.contexts = []
                │
                └─ 7. 清除 event.extra 中的标记

═════════════════════════════════════════════════
最终结果：
  system_prompt = 人格 + 其他插件内容（无 LTM 历史）
  prompt        = 插件格式化的完整上下文
  contexts      = []（空）
```

## 为什么用"短消息"调用再替换？

你可能会问：既然最终要替换成完整上下文，为什么不一开始就直接传完整内容？

这是因为 `event.request_llm()` 调用后会触发 Hook 链，其中一些插件（如 LivingMemory 记忆插件）需要**短消息**来做向量检索：

```
event.request_llm(prompt = "你们觉得去哪好？")   ← 短消息
        ↓
LivingMemory Hook（优先级 0）：
  用 "你们觉得去哪好？" 做向量搜索
  找到相关记忆，追加到 system_prompt
        ↓
本插件 Hook（优先级 -1）：
  LivingMemory 追加的记忆 → 被保留在 system_prompt 中 ✅
  替换 prompt = 完整上下文  ← 此时再换成完整的
```

如果一开始就传完整上下文（几百条历史消息），LivingMemory 会用整个历史去做向量搜索，效果很差而且浪费算力。

## 为什么必须开启平台的"群聊上下文感知"？

> 这是最容易让人困惑的问题：既然本插件已经完全接管了上下文构建，为什么还要在 AstrBot 平台设置里开启**聊天记忆增强（群聊上下文感知）**？

### 原因：插件需要读取平台存储的历史消息

本插件的上下文来源分两部分：

```
┌──────────────────────────────────────────────────────────────┐
│              插件的历史消息读取策略                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  优先级 1：读取平台官方存储                                  │
│  ┌────────────────────────────────────────────────┐         │
│  │ platform_message_history 表（SQLite 数据库）    │         │
│  │                                                 │         │
│  │ • 由 AstrBot 平台的 LTM 系统写入              │         │
│  │ • 数据持久化，重启后不丢失                     │         │
│  │ • 包含完整的群聊消息记录                       │         │
│  │ • 插件通过 context.message_history_manager     │         │
│  │   读取这个表中的数据                           │         │
│  └────────────────────────────────────────────────┘         │
│         ↑                                                    │
│         │  如果读取成功 → 使用这个                           │
│         │  如果读取失败 ↓                                    │
│                                                              │
│  优先级 2：回退到插件自定义存储                              │
│  ┌────────────────────────────────────────────────┐         │
│  │ custom_storage（插件自己的 JSON 文件）          │         │
│  │                                                 │         │
│  │ • 由插件自己写入和管理                         │         │
│  │ • 作为备用数据源                               │         │
│  │ • 受 custom_storage_max_messages 限制          │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**关键点**：`platform_message_history` 表的数据是由平台的 LTM 系统（聊天记忆增强）在 `on_message` 事件中写入的。**如果你不开启"群聊上下文感知"，平台就不会把群聊消息写入这个表，插件就读不到历史消息。**

### 平台 LTM 的双存储机制

AstrBot 平台的 LTM 系统实际上有**两套独立的存储**：

```
┌─────────────────────────────────────────────────────────────────┐
│         AstrBot 平台 LTM 系统的两套存储                         │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  存储 A：内存中的 session_chats（临时）                         │
│  ┌──────────────────────────────────────────────────┐           │
│  │ dict: { "平台:群聊:群号": ["消息1", "消息2", ...] }│          │
│  │                                                    │          │
│  │ • 纯内存，重启后清空                              │          │
│  │ • 受 group_message_max_cnt 限制（默认 300）       │          │
│  │ • 格式：[发送者/HH:MM:SS]: 消息内容              │          │
│  │ • 用途：on_llm_request 中注入到 system_prompt     │          │
│  │         "You are now in a chatroom..."            │          │
│  │                                                    │          │
│  │ ★ 本插件在 on_llm_request 中会删除这部分 ★      │          │
│  │   （因为插件用自己的格式化历史替代了它）          │          │
│  └──────────────────────────────────────────────────┘           │
│                                                                  │
│  存储 B：数据库中的 platform_message_history（持久）            │
│  ┌──────────────────────────────────────────────────┐           │
│  │ SQLite 表：data/data_v4.db                        │          │
│  │                                                    │          │
│  │ • 持久化存储，重启后保留                          │          │
│  │ • 无最大条数限制（平台不会自动清理）              │          │
│  │ • 包含完整消息结构（发送者、时间、内容等）        │          │
│  │ • 用途：供其他插件读取群聊历史                    │          │
│  │                                                    │          │
│  │ ★ 本插件从这里读取历史消息 ★                    │          │
│  │   然后按自己的格式重新组装                        │          │
│  └──────────────────────────────────────────────────┘           │
│                                                                  │
│  两套存储互相独立，不同步                                       │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

所以实际上发生的事情是：

1. 平台 LTM 收到消息 → 写入 session_chats（内存）+ platform_message_history（数据库）
2. 本插件需要回复时 → 从 platform_message_history（数据库）读取历史
3. 平台 LTM 在 on_llm_request 中注入 session_chats 内容到 system_prompt
4. 本插件在 on_llm_request 中**删除**平台注入的内容，**替换**成自己格式化的历史

**一句话总结：插件需要平台的 LTM 帮它"收集"消息并存到数据库，但不需要平台帮它"注入"上下文。收集是收集，注入是注入——收集靠平台，注入靠自己。**

### group_message_max_cnt 与插件的关系

| 配置项 | 属于 | 影响 |
|--------|------|------|
| 平台 `group_message_max_cnt`（默认 300） | AstrBot 聊天记忆增强 | 只限制**内存中 session_chats** 的条数。本插件会删掉这部分内容，所以**对本插件无直接影响** |
| 插件 `max_context_messages`（默认 -1） | 本插件 | 限制插件从数据库读取后**传给 AI 的最大条数** |
| 插件 `custom_storage_max_messages`（默认 500） | 本插件 | 限制插件**自定义备用存储**的最大条数 |

**所以 `group_message_max_cnt` 设置多少都无所谓？**

不完全是。虽然这个值不影响插件读取的历史条数（插件读的是数据库，不受这个限制），但它影响的是平台 LTM 的**内存占用**。设置太大会浪费内存（因为本插件用不上这些内存中的数据），设置太小也没问题。

**推荐设置：**

```
group_message_max_cnt: 10 ~ 50
```

理由：
- 这个值只影响平台自己的内存缓存（session_chats），本插件会删掉它注入的内容
- 设成较小值（如 10-50）可以**节省内存**，因为本插件不依赖它
- 不要设成 0 或关闭，否则平台 LTM 可能不写入 platform_message_history 表
- 保留少量条数是为了确保 LTM 系统正常运行并持续向数据库写入消息

## 为什么必须开启"自动理解图片"？

### 平台的图片处理流程

当群聊中有人发送图片时，AstrBot 平台的 LTM 系统会把消息存入 `platform_message_history`。问题是：**图片怎么存？**

```
                  群友发了一张图片
                        ↓
              平台 LTM handle_message()
                        ↓
              ┌─────────┴─────────┐
              ↓                   ↓
    自动理解图片: 关闭      自动理解图片: 开启
              ↓                   ↓
    存入数据库的内容：       存入数据库的内容：
    "[Alice/14:23:45]:      "[Alice/14:23:45]:
     [Image]"                [Image: 一只橘猫
                              趴在桌子上睡觉]"
              ↓                   ↓
    插件读取到的是：         插件读取到的是：
    "[Image]"               "[Image: 一只橘猫
     ↓                       趴在桌子上睡觉]"
    AI 只知道"有图"          ↓
    但不知道图里是什么       AI 知道图片内容！
```

**关键点**：平台的"自动理解图片"功能会在 LTM 存储消息时，调用 AI 对图片进行描述，并把描述文本存入 `platform_message_history`。当本插件后续从数据库读取历史时，就能拿到带图片描述的消息。

### 那本插件自己的图片处理呢？

本插件有自己的 `enable_image_processing` / `image_to_text_provider_id` 图片处理功能，但两者作用在**不同的时间点**：

```
┌──────────────────────────────────────────────────────────────┐
│            两套图片处理的作用时机对比                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  时机 1：消息被存入历史时（平台"自动理解图片"）             │
│  ┌────────────────────────────────────────────────┐         │
│  │ 群友发图 → 平台 LTM → 调用 AI 描述图片       │         │
│  │         → 存入 "[Image: 描述文本]" 到数据库   │         │
│  │                                                │         │
│  │ 影响：历史消息中的图片有没有描述               │         │
│  │ 没开启 → 历史中只有 "[Image]"，AI 看不懂     │         │
│  │ 开启   → 历史中有 "[Image: 描述]"，AI 能理解 │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
│  时机 2：当前消息被处理时（插件"图片处理"）                 │
│  ┌────────────────────────────────────────────────┐         │
│  │ 群友发图 → 插件概率筛选通过                   │         │
│  │         → 插件调用 AI 描述图片                │         │
│  │         → 把描述嵌入当前消息传给 AI           │         │
│  │                                                │         │
│  │ 影响：当前这一条消息中的图片有没有描述         │         │
│  │ 只在插件决定处理当前消息时才会调用             │         │
│  └────────────────────────────────────────────────┘         │
│                                                              │
│  总结：                                                      │
│  • 平台的 → 负责让 "历史消息" 中的图片有描述               │
│  • 插件的 → 负责让 "当前消息" 中的图片有描述               │
│  • 两者互补，不冲突                                         │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 如果只开插件的图片处理，不开平台的？

历史消息中所有图片都会变成 `[Image]`（无描述），AI 在回复时看不懂历史中的图片内容。当前消息的图片可以处理，但上下文中的历史图片信息会丢失。

### 如果只开平台的，不开插件的？

历史消息中的图片会有描述（好），但当前触发回复的这条消息中的图片可能没有被处理（取决于插件的 `image_to_text_scope` 配置和概率筛选结果）。

### 推荐做法

**两个都开启**，各司其职：

| 设置 | 推荐值 | 说明 |
|------|--------|------|
| 平台 `image_caption`（自动理解图片） | **开启** | 确保历史消息中的图片有描述 |
| 平台 `image_caption_provider_id` | 配置一个轻量模型 | 用于自动描述历史图片 |
| 插件 `enable_image_processing` | **开启** | 确保当前消息中的图片有描述 |
| 插件 `image_to_text_provider_id` | 配置一个视觉模型 | 用于描述当前消息的图片 |

> 两者可以使用同一个 AI 提供商，也可以分别配置不同的模型（比如平台用便宜的，插件用精确的）。

## 推荐的平台设置

综合以上分析，使用本插件时，AstrBot 平台的"聊天记忆增强"应该这样配置：

| 平台配置项 | 推荐值 | 原因 |
|-----------|--------|------|
| 群聊上下文感知 | **开启** | 插件需要平台把群聊消息写入 platform_message_history 数据库 |
| 最大消息数 (`group_message_max_cnt`) | `10` ~ `50` | 只影响平台内存缓存（本插件会删掉注入的内容），设小节省内存 |
| 自动理解图片 (`image_caption`) | **开启** | 让历史消息中的图片有文字描述 |
| 群聊图片转述模型 | 配置一个视觉模型 | 用于生成图片描述文本 |
| 主动回复 (`active_reply`) | **关闭** | 本插件有自己的主动对话系统，两者会冲突 |

## 记忆插件的兼容性：为什么要用适配过的记忆插件？

### 两次 AI 调用，两条完全不同的路径

回顾前文，本插件有**两次独立的 AI 调用**：

```
┌─────────────────────────────────────────────────────────────────┐
│                                                                  │
│  第 1 次调用：决策 AI（"要不要回复？"）                         │
│  ┌──────────────────────────────────────────┐                   │
│  │                                          │                   │
│  │  调用方式：provider.text_chat()          │                   │
│  │                                          │                   │
│  │  ★ 直接调用 AI 提供商 API ★            │                   │
│  │  ★ 完全绕过 Hook 链 ★                  │                   │
│  │  ★ 任何外部插件都无法注入内容 ★        │                   │
│  │                                          │                   │
│  └──────────────────────────────────────────┘                   │
│                                                                  │
│  第 2 次调用：回复 AI（"回复什么内容？"）                       │
│  ┌──────────────────────────────────────────┐                   │
│  │                                          │                   │
│  │  调用方式：event.request_llm()           │                   │
│  │                                          │                   │
│  │  经过 Hook 链 → 外部插件可以注入        │                   │
│  │  但只能注入到 system_prompt              │                   │
│  │  （contexts 和 prompt 会被本插件替换）   │                   │
│  │                                          │                   │
│  └──────────────────────────────────────────┘                   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 未适配的记忆插件：只有回复 AI 能看到记忆

假设你安装了一个本插件**未适配**的记忆插件（不是 LivingMemory 也不是 Legacy），它通常通过 `on_llm_request` Hook 向 `system_prompt` 中注入记忆内容。

那么会发生什么？

```
场景：使用未适配的记忆插件
══════════════════════════════════════════════════

  决策 AI 调用（provider.text_chat）
  ┌───────────────────────────────────────┐
  │                                       │
  │  system_prompt: 人格设定              │
  │  prompt: 群聊上下文 + 当前消息        │
  │                                       │
  │  记忆：❌ 没有                        │
  │                                       │
  │  原因：provider.text_chat() 直接调用  │
  │  AI 提供商，完全不经过 Hook 链，      │
  │  外部记忆插件根本没有机会注入任何     │
  │  内容                                 │
  │                                       │
  │  后果：AI 在判断"要不要回复"时        │
  │  完全不知道之前跟这个人聊过什么，     │
  │  无法基于历史关系做出更准确的判断     │
  │                                       │
  └───────────────────────────────────────┘
              ↓ 决策 AI 说"要回复"

  回复 AI 调用（event.request_llm）
  ┌───────────────────────────────────────┐
  │                                       │
  │  system_prompt:                       │
  │    人格设定                           │
  │    + 外部记忆插件注入的记忆 ✅        │
  │    （通过 Hook 链注入到 system_prompt │
  │      本插件会保留这部分内容）         │
  │                                       │
  │  prompt: 插件格式化的完整上下文       │
  │                                       │
  │  记忆：✅ 有（在 system_prompt 中）   │
  │                                       │
  └───────────────────────────────────────┘
```

**结论：未适配的记忆插件只能影响"回复内容"，无法影响"是否回复"的决策。**

### 适配过的记忆插件：两次 AI 调用都能看到记忆

当使用本插件适配的记忆插件（LivingMemory 或 Legacy），并开启 `enable_memory_injection` 时：

```
场景：使用适配过的记忆插件 + enable_memory_injection = true
══════════════════════════════════════════════════

  插件主动调用记忆插件 API 获取记忆
  ┌───────────────────────────────────────┐
  │ MemoryInjector.get_memories()         │
  │                                       │
  │ 直接调用记忆插件的内部接口            │
  │ 不依赖 Hook 链                        │
  │ 拿到记忆文本后自己嵌入上下文          │
  └───────────────────────────────────────┘
              ↓ 拿到记忆文本

  ┌─────────────────── memory_insertion_timing 配置 ──────────────┐
  │                                                                │
  │  "pre_decision"（决策前注入）          "post_decision"（决策后）│
  │  ┌──────────────────────┐             ┌──────────────────────┐│
  │  │                      │             │                      ││
  │  │ 记忆嵌入到上下文中   │             │ 记忆在决策后才嵌入   ││
  │  │      ↓               │             │                      ││
  │  │ 决策 AI ✅ 有记忆   │             │ 决策 AI ❌ 无记忆   ││
  │  │      ↓               │             │      ↓               ││
  │  │ 回复 AI ✅ 有记忆   │             │ 回复 AI ✅ 有记忆   ││
  │  │ （复用决策阶段的     │             │ （此时才注入记忆）   ││
  │  │  上下文缓存）        │             │                      ││
  │  │                      │             │                      ││
  │  └──────────────────────┘             └──────────────────────┘│
  │                                                                │
  └────────────────────────────────────────────────────────────────┘
```

### 完整对比

| 场景 | 决策 AI 有记忆？ | 回复 AI 有记忆？ | 记忆注入方式 |
|------|:---:|:---:|------|
| 未适配的记忆插件 | ❌ | ✅ | 外部插件通过 Hook 注入 system_prompt |
| 适配插件 + `post_decision` | ❌ | ✅ | 插件主动调用记忆 API，决策后嵌入 |
| 适配插件 + `pre_decision` | ✅ | ✅ | 插件主动调用记忆 API，决策前嵌入 |

### 为什么决策 AI 有没有记忆很重要？

决策 AI 的任务是判断"现在要不要回复这条消息"。在本插件的决策提示词中，明确写了：

> **有记忆时更倾向于回复**，特别是：
> - 消息与记忆内容高度相关
> - 记忆显示与当前发送者有重要互动历史

也就是说，如果决策 AI 有记忆，它能做出更"人性化"的判断：

```
没有记忆的决策 AI：
  "张三说了'还记得上次那件事吗？'" → 不知道什么事 → 可能判断为不回复

有记忆的决策 AI：
  "张三说了'还记得上次那件事吗？'"
  + 记忆显示：上次和张三讨论了一个有趣的话题
  → 这个人跟我聊过！这条消息很相关！→ 判断为应该回复
```

### 为什么外部记忆插件不能注入到决策 AI？

根本原因是**调用方式不同**：

```
决策 AI：
  provider.text_chat(prompt, contexts, system_prompt)
  │
  │  这是直接调用 AI 提供商的底层 API
  │  完全不触发平台的 Hook 机制
  │  外部插件无法感知这次调用的存在
  │
  └→ 所以外部记忆插件的 on_llm_request Hook 根本不会被触发

回复 AI：
  event.request_llm(prompt, contexts, system_prompt)
  │
  │  这是通过平台标准流程调用
  │  会触发完整的 Hook 链
  │  外部插件可以在 Hook 中修改请求
  │
  └→ 外部记忆插件的 Hook 会被触发，可以注入到 system_prompt
```

**这不是设计缺陷，而是有意为之**——决策 AI 需要轻量快速地做判断，如果走 Hook 链，其他插件的修改可能干扰决策逻辑。但这也意味着，只有本插件**主动适配**的记忆插件才能把记忆送到决策 AI 面前。

### 推荐做法

| 推荐等级 | 做法 | 效果 |
|:---:|------|------|
| ⭐⭐⭐ | 使用适配的记忆插件 + `pre_decision` | 决策和回复都有记忆，AI 表现最好 |
| ⭐⭐ | 使用适配的记忆插件 + `post_decision` | 回复有记忆，决策无记忆 |
| ⭐ | 使用未适配的记忆插件 | 只有回复有记忆，决策完全无记忆 |

**目前适配的记忆插件：**

| 插件 | 配置模式 | 说明 |
|------|----------|------|
| [strbot_plugin_play_sy](https://github.com/kjqwer/strbot_plugin_play_sy) | `memory_plugin_mode: "legacy"` | 传统模式，稳定性高，**推荐** |
| [astrbot_plugin_livingmemory](https://github.com/lxfight-s-Astrbot-Plugins/astrbot_plugin_livingmemory) | `memory_plugin_mode: "livingmemory"` | 智能模式，混合检索+人格隔离 |
| 自动检测 | `memory_plugin_mode: "auto"` | 自动检测已安装的记忆插件（优先 LivingMemory） |

## 其他常见问题

### Q: 我用了其他"智能回复"插件，为什么 AI 行为异常？

A: 如果另一个插件也修改 `req.contexts` 或 `req.prompt`，两者会互相覆盖。由于本插件优先级最低（-1），它会覆盖其他插件对这两个字段的修改。**建议只使用一个上下文管理类插件。**

### Q: 为什么 AstrBot 的 `/reset` 清除历史后，AI 还记得之前的对话？

A: 因为本插件使用**自己的消息存储**（custom_storage），不依赖平台的 `conversations` 表。`/reset` 只清除平台的表，不影响插件存储。请使用插件的 `gcp_reset` 或 `gcp_reset_here` 指令。

### Q: emotionai / FavourPro / SelfLearning 等插件能和本插件一起用吗？

A: 可以。这些插件只修改 `system_prompt`，本插件会保留它们追加的内容。

### Q: 开启 `enable_debug_log` 后日志中看到的上下文是什么？

A: 是插件最终发送给 AI 的完整内容，包括格式化的历史消息、系统指令、当前消息等。这是排查问题的最佳方式。

### Q: 不开启平台的"群聊上下文感知"会怎样？

A: 插件将无法从 `platform_message_history` 表中读取历史消息，会回退到自定义存储。自定义存储的数据可能不如平台存储完整（特别是在插件刚安装、或者重启后）。**强烈建议开启。**

### Q: `group_message_max_cnt` 设成 300 会不会浪费？

A: 会略微浪费内存，但不会影响功能。平台会在内存中维护 300 条消息的缓存，然后在 on_llm_request 中注入到 system_prompt，但本插件会立即删掉这部分注入的内容。设成 10-50 更节省资源。

### Q: 平台的"自动理解图片"和插件的"图片处理"会重复调用 API 吗？

A: 对于**历史消息**中的图片，只有平台会调用一次（存储时）。对于**当前正在处理的消息**中的图片，如果插件决定处理，会再调用一次。两者作用在不同时机，不会对同一张图片重复调用。

## AI 回复频率相关问题

### Q: AI 回复频率太高，一直在刷屏怎么办？

A: 回复频率过高通常是概率和限制相关配置偏高导致的。按优先级依次排查和调整：

| 排查项 | 配置项 | 建议操作 |
|--------|--------|----------|
| 1. 基础概率过高 | `initial_probability` | 降低到 `0.05` ~ `0.08`（默认 `0.08`），这是没有注意力加成时的基础触发概率 |
| 2. 连续对话概率过高 | `after_reply_probability` | 降低到 `0.5` ~ `0.7`（默认 `0.8`），回复一次后的后续触发概率 |
| 3. 回复密度限制未开启 | `enable_reply_density_limit` | 确保为 `true`，限制短时间内的最大回复次数 |
| 4. 密度窗口/上限 | `reply_density_window_seconds` / `reply_density_max_replies` | 默认 300 秒内最多 4 次，可改为 `300` 秒 / `2~3` 次 |
| 5. 对话疲劳未开启 | `enable_conversation_fatigue` | 确保为 `true`，连续对话后自动降低回复意愿 |
| 6. 时段概率未配置 | `enable_dynamic_reply_probability` | 开启后配置 `reply_time_periods`，深夜/午休时段自动降低频率 |
| 7. 平台主动回复未关闭 | AstrBot 聊天记忆增强 → 主动回复 | **必须关闭**，否则平台和插件会同时回复，频率翻倍 |

**快速降频方案**：将 `initial_probability` 改为 `0.03`，`after_reply_probability` 改为 `0.5`，`reply_density_max_replies` 改为 `2`。

### Q: AI 回复频率太低，基本不说话怎么办？

A: 回复频率过低的排查方向：

| 排查项 | 配置项 | 建议操作 |
|--------|--------|----------|
| 1. 基础概率太低 | `initial_probability` | 提高到 `0.10` ~ `0.15` |
| 2. 是否处于深夜/午休低频时段 | `reply_time_periods` | 检查当前时间是否命中低频时段（`factor < 1.0`），可调大对应 `factor` |
| 3. 对话疲劳阈值过低 | `fatigue_threshold_light` | 提高轻度疲劳阈值，如改为 `5`，让 AI 能连续多聊几轮 |
| 4. 频率调整器持续降频 | `enable_frequency_adjuster` | 暂时关闭观察，或调高 `frequency_min_probability`（如 `0.05`） |
| 5. 决策 AI 总是回答 NO | `decision_ai_provider_id` | 确认决策模型正常工作，开启 `enable_debug_log` 查看决策结果 |
| 6. 群组未启用 | `enabled_groups` | 留空 = 全部启用；如果填了群号，确认目标群在列表中 |
| 7. 注意力机制降低了概率 | `attention_decreased_probability` | 当 AI 对某用户注意力很低时会使用此值，可适当提高 |

**快速提频方案**：将 `initial_probability` 改为 `0.15`，`after_reply_probability` 改为 `0.85`，暂时关闭 `enable_conversation_fatigue`。

### Q: AI 回复频率忽高忽低、不稳定？

A: 这通常是多个动态系统叠加的正常表现。如果波动过大，可以逐一排查：

1. **频率调整器**（`enable_frequency_adjuster`）：每隔一段时间自动分析群聊节奏并调整概率，可能导致突然升降。如果不需要可以关闭
2. **注意力机制**（`enable_attention_mechanism`）：对不同用户有不同注意力值，会影响触发概率。查看日志中注意力分数变化
3. **动态时段概率**（`enable_dynamic_reply_probability`）：在时段切换的过渡期（`reply_time_transition_minutes`）概率会平滑变化
4. **对话疲劳**（`enable_conversation_fatigue`）：连续对话后突然降频是疲劳系统生效，等 `fatigue_reset_threshold` 秒后会恢复

**建议**：开启 `enable_debug_log`，观察日志中各系统对概率的实际影响值。

## AI 读空气判断相关问题

### Q: AI 读空气总是判断为不回复（NO），该怎么调？

A: "读空气"由决策 AI 负责，它会综合上下文判断当前是否应该回复。如果总是 NO：

1. **检查决策 AI 模型**：`decision_ai_provider_id` 所配置的模型是否正常工作？某些轻量模型可能理解力不足，总是保守回答 NO
2. **调整决策提示词**：通过 `decision_ai_extra_prompt` 追加提示，例如：`"你应该更积极地参与对话，当有人在讨论有趣的话题时倾向于回复 YES"`
3. **检查提示词模式**：`decision_ai_prompt_mode` 设为 `"append"` 是追加模式，`"override"` 会完全替换内置提示词（不推荐，除非你很清楚在做什么）
4. **注意力和记忆**：如果用了适配的记忆插件且 `memory_insertion_timing` 设为 `"pre_decision"`，记忆会帮助决策 AI 做出更准确的判断。没有记忆时决策 AI 可能偏保守
5. **概率筛选先行**：读空气判断是第二层筛选，消息必须**先通过概率筛选**才会进入读空气判断。确保 `initial_probability` 不会太低导致大部分消息在概率筛选阶段就被过滤了

### Q: AI 读空气判断太宽松，什么消息都回复？

A: 如果 AI 几乎对所有消息都判断 YES：

1. **检查是否跳过了决策 AI**：@消息、触发关键词等会跳过概率和读空气判断直接回复，确认是否大量消息通过这些旁路触发
2. **调整决策提示词**：通过 `decision_ai_extra_prompt` 追加限制，例如：`"只有当消息明确与你相关、直接提问、或话题非常有趣时才回复 YES，闲聊和水群不需要回复"`
3. **更换决策模型**：某些模型可能倾向于积极回应，尝试换一个更"理性"的模型
4. **利用消息质量预判**：开启 `enable_message_quality_scoring`，纯水聊消息会被降权，减少到达决策 AI 的低质量消息

### Q: 怎么查看 AI 读空气的具体判断过程？

A: 开启 `enable_debug_log`，日志中会显示：
- 概率筛选的计算过程和结果
- 决策 AI 收到的上下文内容
- 决策 AI 的回答（YES / NO）及原因
- 各个系统（注意力、疲劳、时段等）对概率的调整详情

## Web 管理面板相关问题

### Q: Web 管理面板的默认端口是什么？怎么访问？

A: Web 管理面板的相关信息：

| 项目 | 说明 |
|------|------|
| **默认端口** | `1451` |
| **默认监听地址** | `0.0.0.0`（所有网络接口） |
| **访问地址（本机）** | `http://localhost:1451` 或 `http://127.0.0.1:1451` |
| **访问地址（远程）** | `http://你的服务器IP:1451` |
| **默认状态** | 关闭（需在插件配置中将 `enable_web_panel` 设为 `true`） |

**首次使用步骤**：

1. 在 AstrBot 插件配置中将 `enable_web_panel` 设为 `true`
2. 重启 AstrBot（或重载插件）
3. 查看 AstrBot 日志，首次启动会输出**随机生成的默认密码**
4. 浏览器访问 `http://你的IP:1451`，使用默认密码登录
5. 登录后在 Web 面板中修改密码

### Q: Web 面板打不开 / 无法访问怎么办？

A: 按以下步骤逐一排查：

**第一步：确认插件已启用 Web 面板**
- 检查 `enable_web_panel` 是否为 `true`
- 检查 AstrBot 日志中是否有 Web 面板启动成功的信息

**第二步：确认端口和地址**
- 确认 `web_panel_port`（默认 `1451`）没有被其他程序占用
- 如果修改过端口，确认访问的是正确端口

**第三步：检查防火墙**
- **Windows**：检查 Windows Defender 防火墙是否放行了对应端口
  ```
  # 以管理员身份运行 PowerShell，添加入站规则
  New-NetFirewallRule -DisplayName "GCP Web Panel" -Direction Inbound -Port 1451 -Protocol TCP -Action Allow
  ```
- **Linux**：检查 iptables / firewalld / ufw 是否放行
  ```bash
  # ufw
  sudo ufw allow 1451/tcp

  # firewalld
  sudo firewall-cmd --permanent --add-port=1451/tcp && sudo firewall-cmd --reload
  ```
- **云服务器**（阿里云/腾讯云/华为云等）：除了系统防火墙，还需在**云控制台的安全组**中放行对应端口

**第四步：Docker 部署额外检查**
- 确认容器启动时映射了端口：`-p 1451:1451`
- 如果使用 docker-compose，检查 `ports` 配置：
  ```yaml
  ports:
    - "1451:1451"
  ```
- 如果修改了 `web_panel_port`，映射也要对应修改：`-p 你的端口:你的端口`
- 确认容器内部监听地址为 `0.0.0.0`（默认值），而不是 `127.0.0.1`（`127.0.0.1` 在容器内只允许容器内部访问）

**第五步：反向代理检查**（如果使用 Nginx 等反代）
- 确认反代配置正确转发到 `127.0.0.1:1451`
- 如果使用反代，建议开启 `web_panel_trust_proxy` 以正确获取客户端真实 IP

### Q: 忘记了 Web 面板密码怎么办？

A: 两种方式重置密码：

1. **通过配置重置**：在 AstrBot 插件配置中将 `web_panel_reset_password` 设为 `true`，然后重启。密码会被重置为新的随机值并输出到日志中。重置完成后此开关会自动关闭
2. **删除密码文件**：删除插件数据目录下的 `web_panel_password.json` 文件，重启后会重新生成随机密码并输出到日志

### Q: Web 面板提示"IP 已被封禁"怎么办？

A: 多次输入错误密码会触发暴力破解防护（分级锁定）：

| 失败次数 | 锁定时长 |
|:--------:|:--------:|
| 5 次 | 30 秒 |
| 10 次 | 60 秒 |
| 15 次 | 5 分钟 |
| 20 次 | 10 分钟 |

等待锁定时间过后即可重新尝试。如果确实忘记密码，请参考上面的密码重置方法。
